How Your App

No React knowledge needed. Diagrams, analogies, and plain English only.

What this guide does: It shows you the skeleton of your app — how the pieces connect, why they exist, and what each one does. After reading this you will be able to explain your own architecture to your mentor with confidence.

The Simplest Possible Explanation

Analogy — Think of it like a restaurant
The customer (user) sits at a table and sees only the menu and their plate.
The waiter (your React app) takes the order, shows results, and talks to the kitchen.
The kitchen (Firebase) stores all the food (data) and processes orders.

The customer never goes into the kitchen. The kitchen doesn't talk to the customer directly. The waiter (your app) is always in the middle.

Key point: Your app (React) lives in the browser — not on a server you built. Firebase is the only server involved, and Google runs it for you. You wrote zero backend code.
"The app has three layers. The user interacts with a React interface that runs in the browser. When data needs to be saved or retrieved — like a prescription or a user's role — the app talks to Firebase, which is Google's cloud service. Firebase handles both login security and the database."

The 3 Layers of Your App

Every web application has layers — like a building has floors. Your app has three:

Layer 1

What the User Sees — The UI Layer

All the pages, buttons, forms, cards, and dashboards. This is built with React (the components) and styled with Tailwind CSS (the colours and layout). Think of this as the paint, furniture, and windows of a building.

↓
Layer 2

The Logic Layer — How the App Thinks

The rules that govern behaviour. Who is logged in? What role do they have? Which page should they see? What happens when a form is submitted? This is handled by React Context (global state), React Router (navigation), and the RequireRole guard (access control). Think of this as the electrical wiring and plumbing inside the walls.

↓
Layer 3

Where Data Lives — The Data Layer

All the real information: user accounts, prescriptions, medicines, inventory, orders. This is stored in Firebase Firestore (the database) and protected by Firebase Authentication (login system). Think of this as the basement vault where everything important is kept.

"The app is structured in three layers. The top layer is the UI — what the user sees, built with React and styled with Tailwind. The middle layer is the logic — routing, role checking, global state. The bottom layer is Firebase — authentication and the Firestore database where all data is stored."

The Role System — Who Sees What

Analogy — Think of it like a hotel key card
When you check into a hotel, your room key only opens your room and shared areas (gym, pool). It does not open other guests' rooms or the staff office. Your role in MedFind works the same way — it unlocks only the pages that belong to you.
WHO GETS WHAT
Doctor

Dashboard · New Prescription · Patient List · Pharmacy List · Prescriptions · Connections · Search · Settings

Patient

Dashboard · My Prescriptions · Dose Schedule · My Doctors · Search · Settings

Pharmacist

Dashboard · Inventory · Medicines · Orders · Prescriptions · Search · Settings

Admin

Dashboard · Users · Doctors · Patients · Pharmacies · Prescriptions · Search · Settings

How does the app know which role the user has?

1
User signs up and picks a role
When creating an account, they choose: Doctor, Patient, or Pharmacist. This role is saved in Firebase.
2
On login, the app fetches the profile
Firebase confirms the password. Then the app goes to Firestore and gets the user's profile — which includes their role.
3
Role is stored in the global notice board (Context)
The role is placed in React Context — a shared memory space. Now every part of the app can read "this user is a Doctor."
4
The correct dashboard and menu items appear
The sidebar reads the role and shows only the relevant links. The wrong role's pages are never even rendered.
"When a user logs in, the app fetches their profile from Firestore which contains their role — doctor, patient, or pharmacist. This role is stored in a React Context so every component in the app can read it. The sidebar and the navigation then show only the menu items that belong to that role."

The Security Guard — How Pages Are Protected

Analogy — Think of it like a bouncer at a club
Before you enter any room, a bouncer checks your ID (your role). If your ID says you're allowed in, you pass. If not, you're turned away — even if you know the address. In MedFind this bouncer is a piece of code called RequireRole.
User navigates to a page
(or types a URL manually)
↓
RequireRole checks:
"Is this user's role allowed here?"
↓
↙
YES — role matches
✓ Page loads normally
↘
NO — wrong role
✗ Sent to Unauthorized page

Example — What happens if a Patient tries to access the Prescription form?

!
Patient types /dashboard/new-prescription in the browser
This is a Doctor-only page.
→
RequireRole runs instantly
It reads the role from Context: "patient". The allowed list for this page is: ["doctor"]. No match found.
✓
Patient is redirected to Unauthorized page
The prescription form is never shown. The data is also protected by Firebase rules — so even a direct database request would be blocked.
"Every protected route in the app is wrapped by a component called RequireRole. It checks the user's role from the global context before rendering any page. If the role doesn't match, the user is immediately redirected to an Unauthorized page — even if they manually type the URL. On top of that, Firebase security rules protect the database directly, so a clever user can't bypass the frontend and query the data directly either."

Firebase — Your Ready-Made Backend

Analogy — Think of Firebase like a fully-equipped kitchen
You didn't build the kitchen from scratch — you rented one that already has ovens, refrigerators, and a storage room. You just put your ingredients (data) in it and use the appliances (authentication, database). Firebase is Google's kitchen.

Firebase Authentication

  • Manages email + password login
  • Stores passwords securely (hashed)
  • Issues a token when user logs in
  • Tells the app: "User is logged in / out"
  • Handles sign up, sign in, sign out

How a Page Loads — The Full Journey

Let's trace exactly what happens when a Doctor logs in and opens the New Prescription page. This is the complete picture of your architecture in action.

1
Doctor opens the browser and goes to the app URL
The browser downloads your React app. At this point, no one is logged in. The app shows the Login page.
2
Doctor types email + password and clicks "Sign In"
The app sends the credentials to Firebase Authentication. Firebase checks them and says "OK".
3
App fetches the doctor's profile from Firestore
Using the user ID from Firebase Auth, the app reads the profile document from the "profiles" collection. It gets: name, role = "doctor", verified = true.
4
Profile is saved to the global notice board (Context)
Every component in the app can now read: "This is Dr. Ahmed. Role: doctor."
5
Router redirects the doctor to /dashboard
The router reads the role and sends the doctor to their dashboard. The sidebar shows Doctor-specific menu items.
6
Doctor clicks "New Prescription" in the sidebar
The router tries to load the New Prescription page. But first, RequireRole runs.
7
RequireRole checks: "Is 'doctor' in the allowed list?"
Allowed list for this page: ["doctor"]. Role is "doctor". ✓ Pass.
8
Page code is downloaded (lazy loading)
Because this page uses lazy loading, its JavaScript file only downloads now — not when the app first loaded. A spinner shows for a moment.
9
Page fetches its data from Firestore
The New Prescription page needs: the doctor's patients (from "connections" collection) and the medicine list (from "medicines" collection). These are fetched from Firestore.
✓
The form appears with the doctor's data pre-loaded
Patient dropdown shows connected patients. Medicine dropdown shows the medicines database. The doctor can now write a prescription.

Key Terms — In Plain English

These are the words your mentor might use. Know what they mean — not how to spell them in code.

TermPlain English Meaning
Component A reusable piece of the UI. Like a LEGO brick. A button is a component. A card is a component. A page is made of many components.
Page What shows when you go to a specific URL. Made of components. Example: the New Prescription page is one big component made of smaller form components.
Route The rule that says: "When the URL is /dashboard/prescriptions, show the Prescriptions page." Router.jsx contains all the routes.
State Data that can change. Example: the loading spinner is shown while loading = true and hidden when loading = false. State is what makes the UI react to things happening.
Context A global notice board that every component can read. Instead of passing the user's role through every component manually, you put it in Context once and read it anywhere.
Props Data passed from a parent component to a child. Like giving instructions when you ask someone to do a job: "Here is the list of patients — show them in a dropdown."
Hook A special React function that lets a component do something — like remember data (useState), fetch data when the page loads (useEffect), or read from the global notice board (useContext).
Lazy Loading Only download the code for a page when the user actually visits it. Keeps the app fast on first load.
Guard / RequireRole A component that checks your role before letting you into a page. Like a bouncer with a guest list.
Firestore Firebase's database. Stores data as "documents" inside "collections" (like files inside folders). Everything in your app — prescriptions, profiles, medicines — is a document.
Authentication "Who are you?" — Verifying the user's identity via email and password login.
Authorization "What are you allowed to do?" — Checking the user's role to decide which pages and data they can access.
NoSQL A type of database that stores flexible documents instead of rigid rows and columns. Firestore is NoSQL.
Async / Await When the app needs to wait for something (like Firebase to reply), it uses async/await. The rest of the app keeps working while it waits — it does not freeze.

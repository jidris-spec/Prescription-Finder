The 3 Most Important Concepts to Understand

These are the concepts your mentor will almost certainly ask about. Understand what they mean — not the code.

Concept A — Role-Based Access Control (RBAC)
What it means: Different users see different things based on their role. A doctor can write prescriptions. A patient cannot. A pharmacist can manage inventory. Others cannot. This is enforced at two levels:

1. Route level — A component called RequireRole wraps every page. Before showing the page, it checks the user's role. Wrong role = redirected away.

2. Database level — Firebase security rules on Firestore prevent users from reading or writing data they don't own.
"I built a role guard component that wraps every protected route. When a user navigates to a page, the guard checks their role from the global auth context. If the role doesn't match what the page allows, they're sent to an Unauthorized page. This works even if someone manually types the URL — they still can't get in."
Concept B — React Context (Global State)
What it means: When a user logs in, the app needs to know: Who is this person? What is their role? This information needs to be available everywhere in the app — the sidebar, the dashboard, the route guard.

React Context is like a shared notice board the whole app can read. After login, the user's info and role are pinned to that board. Every component reads from it.
"I use React Context to store the logged-in user's profile and role. After Firebase authenticates the user, I fetch their profile from Firestore and store it in context. Then every component in the app — the sidebar, the role guard, the dashboard — can access this information without me passing it through props."
Concept C — Lazy Loading
What it means: Instead of downloading the entire app when the page first loads, only the minimum necessary code is loaded. The doctor's prescription pages, the patient's schedule, the pharmacy inventory — these are separate "chunks" that only download when the user navigates to them.

This makes the initial load faster, especially important for users on slower connections.
"I used React's lazy() and Suspense to split the app into chunks. Each role's pages are only downloaded when someone visits them. If a patient logs in, they never download the doctor's or pharmacy's code. This keeps the initial load fast."

Each step explained simply
Step 1 — Connection: A doctor can only prescribe for patients they are connected to. The patient sends a request, the doctor accepts. This prevents random prescriptions.

Step 2 — The form: The doctor picks the patient, types the diagnosis, then adds medicines. Each medicine has a name, dosage (e.g. 500mg), how often (e.g. twice daily), and for how long (e.g. 30 days). They can add multiple medicines in one prescription.

Step 3 — Saving: When submitted, a prescription record is saved to Firestore. Each medicine gets its own item record linked to that prescription. The expiry date is calculated automatically.

Step 4 — Patient view: The patient can see all their prescriptions and a daily dose schedule based on what medicines they are taking.

Step 5 — Pharmacy: Pharmacists see incoming orders, check their stock, and mark orders as filled.
If asked: "What validation does the prescription form have?"
"The form checks several things before saving: the doctor must be verified by the admin, a patient must be selected, and at least one medicine must be added with a dosage and frequency. If any of these fail, a clear error message shows and nothing is saved."

How to Handle Tough Questions

Remember: Your mentor is not trying to trick you. They ask hard questions to see how you think, not to see if you've memorised everything. Honest, thoughtful answers are always better than guessing.
"What would you improve if you had more time?"
"I'd add real-time notifications — so the patient gets an alert the moment a prescription is created. I'd also add PDF export for prescriptions, proper automated testing, and eventually a mobile version. The current code structure makes these additions straightforward since each role is isolated in its own folder."
"What was the hardest part to build?"
"The role-based routing was the trickiest part. I needed every page to be protected, but the user's role only becomes available after an async Firebase call. I had to make sure the loading state was handled correctly — so the app doesn't flash an Unauthorized page before the profile is even loaded."
"What did you learn from this project?"
"I learned how to think architecturally — how to organise a real app so it stays manageable as it grows. I also learned how authentication and authorization are different things: authentication is 'who are you?', authorization is 'what are you allowed to do?' Both are handled differently in this project."
"I don't know the answer to this" — what to say
"That's a great question — I haven't explored that part yet. Can I note it down and come back to you after I've researched it?"

Authentication vs Authorization

Authentication = "Who are you?" (login with email/password)
Authorization = "What can you do?" (role guard checks your role)

Frontend vs Backend

Frontend = what the user sees (React, Tailwind, your code)
Backend = where data is stored (Firebase — you didn't build this yourself)

Component vs Page

Component = a reusable piece of UI (a button, a card, a form)
Page = what shows for a URL, usually made of multiple components

State vs Props

State = data a component manages itself (e.g. loading, error)
Props = data passed into a component from outside (e.g. the list of medicines)

The Demo Plan — 12 Minutes, Step by Step

Before the meeting: Run npm run dev. Create test accounts in advance. Have a sample prescription already saved so there is real data to show.
  • 1️⃣
    Login page (1 min)
    Show the sign-in form. Say: "Firebase handles authentication here — I don't store passwords, Firebase does that securely." Log in as a doctor.
  • 2️⃣
    Doctor dashboard (2 min)
    Show the welcome screen and sidebar. Say: "The sidebar only shows pages this role can access. A patient logging in would see a completely different menu."
  • 3️⃣
    New Prescription page (3 min) — your star feature
    Walk through the form slowly. Select a patient. Add a medicine with dosage and frequency. Submit. Show the success redirect. Say: "The prescription is now in Firestore and the patient can see it immediately."
  • 4️⃣
    Switch to patient account (2 min)
    Log in as a patient. Open the prescriptions list — the prescription you just wrote is there. Then show the dose schedule. Say: "Same data, different perspective."
  • 5️⃣
    Switch to pharmacy account (2 min)
    Show the inventory dashboard with the stats cards. Open the orders list. Say: "The pharmacist sees prescriptions as orders to fulfill."
  • 6️⃣
    Show the role guard (1 min)
    While logged in as a patient, type /dashboard/new-prescription in the URL. The Unauthorized page shows. Say: "The role guard blocks this even if you know the URL."
  • 7️⃣
    Open to questions (1 min)
    Say: "That covers the main features. I'd love your feedback — what's the first thing you would improve?"
